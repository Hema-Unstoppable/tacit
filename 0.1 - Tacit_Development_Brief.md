# Tacit — Development Brief
**Working title: Tacit** (final naming decision pending — see Section 2)

## How to use this document

This is a development brief, written to be handed to Claude Code as the starting context for building this tool. It covers the product concept, branding, user experience, data architecture, integrations, and a recommended coding structure. Sections marked **[DECISION NEEDED]** are places where a choice should be confirmed before or during the build — Claude Code should ask rather than assume on these.

This is **not** a multi-tenant SaaS product. It is a personal tool, built for one user (Ibrahim Awad), managing two LinkedIn destinations he personally controls: his own profile and the TOGA Academy company page. There is no billing, no customer accounts, no OAuth flow for anyone other than the owner. Keeping this boundary clear matters — it's what keeps the build small enough to finish.

---

## 1. Project vision

Ibrahim has sixteen-plus years of engineering documents, procedures, and frameworks sitting in a Google Drive folder, and two LinkedIn destinations (his personal profile and the TOGA Academy page) that need consistent, high-quality activity. The manual work of mining a document for what's actually interesting in it, writing it up in a voice that sounds like him, and remembering to post on a schedule, is the actual bottleneck — not a lack of expertise or a lack of ideas.

Tacit automates everything between "I have a document, a topic, or a campaign to run" and "a draft is sitting in my queue, written in my voice, ready for a one-click approval." Nothing gets published without that human approval step. The tool's job is to do the first 90% of the work; Ibrahim's job is the last 10% — the judgment a generic AI tool can't replicate, which is also the entire thesis behind why this is worth building instead of just using an off-the-shelf tool.

**What this is not:** a lead-generation tool, a LinkedIn outreach/connection-automation tool, or a multi-account SaaS product. Those are different problems with different (and riskier) solutions, and bolting them on would dilute both the build and the brand.

That's both an IP-safety matter and a genuine product decision: Tacit should look like it belongs to TOGA, not like an unbranded Taplio clone.

## 3. Feature parity with Taplio — what to replicate, adapt, or skip

Taplio is, at its core, four things: AI content generation trained on the user's voice, a visual content calendar, an engagement analytics dashboard, and (at its higher tiers) a sales lead database with outreach automation. The fourth one is the part to actively avoid — it's a different problem (cold outreach, not content), and some third-party LinkedIn automation tools in this space rely on browser-cookie-based authentication that violates LinkedIn's terms of service and carries real account-suspension risk. Tacit only ever talks to LinkedIn through the official, documented Posts API with proper OAuth — never through session hijacking, scraping, or browser automation pretending to be a human.

| Taplio feature | Decision | Notes |
|---|---|---|
| AI writer trained on your voice | **Replicate** | Core feature — see Section 8 (voice profiles) |
| Visual content calendar | **Replicate** | One calendar, color-coded by channel |
| Multi-account management | **Replicate** | Personal profile + TOGA page, not arbitrary multi-tenant accounts |
| Post inspiration library (viral post browsing) | **Adapt** | Not a library of *other people's* viral posts — Tacit's "inspiration" comes from Ibrahim's own document library and topic/news input instead |
| Engagement analytics | **Adapt** | LinkedIn's API gives limited analytics outside enterprise partner tiers — v1 analytics will mostly be manually logged numbers, not a live-pulled dashboard |
| Carousel / video content generation | **Skip for v1** | Real feature, real future option, not core to the current pain |
| Lead database / contact enrichment | **Skip entirely** | Different problem, ToS risk, scope creep |
| Auto-commenting / connection automation | **Skip entirely** | Same reasons as above |

## 5. User experience flows

**A — First-time setup**
1. Open the app, log in with the single password.
2. Connect both LinkedIn destinations (personal profile, TOGA page) through OAuth.
3. Paste in 15–20 of your best past posts per channel to seed the voice profile.
4. Set the content-mix ratio (default: 1 promotional post for every 3 organic/technical posts).

**B — Add a document and get drafts**
1. Drop a file into the Sources screen (or it appears automatically if the Drive-sync folder option is enabled).
2. The confidentiality screen runs automatically; if flagged, the document sits in "needs your review" status until cleared.
3. Once cleared, the document is parsed, insights are extracted, and three post-variant drafts appear in the Dashboard queue within roughly 30 seconds.

**C — Train or refresh a voice profile**
1. Go to Voice Profiles, pick a channel.
2. Paste in reference posts (or add new ones to an existing set).
3. Click "Train" — one Claude call extracts a structured style fingerprint, shown back to you in plain language for sanity-checking.

**D — Launch a campaign**
1. Go to Campaigns → New Campaign.
2. Pick a product (or create one), pick a channel, pick a length (default 5 posts over 2 weeks).
3. Claude generates a sequenced draft set with a narrative arc (teaser → value → proof → urgency → recap) — proof-style posts never include a fabricated testimonial; if you haven't supplied a real one, that slot becomes a generic, unattributed claim instead.
4. Review each post in the queue exactly like any other draft.

**E — Daily review habit**
1. Open the Dashboard. See pending drafts, today's scheduled posts, and anything flagged for confidentiality review.
2. Edit, pick a variant, approve and assign a date, or reject.
3. Nothing posts without this step, ever.

**F — Resolve a confidentiality flag**
1. Dashboard shows a flagged source with the reason Claude gave (e.g., "mentions a named client project").
2. Either edit the source text to remove the sensitive detail, or manually mark it cleared if it's a false positive.

## 6. UI/UX design guidelines

- **The draft is the centerpiece, not the AI.** Drafts are shown large and immediately editable. AI involvement is visually present (a small "AI-drafted" tag) but never dominant — this is a writing assistant, not a content slot machine.
- **Status is always visible, always color-coded:** Draft (gray), Needs review (amber), Scheduled (teal), Posted (a muted green), Rejected (a muted red). Never rely on text alone to communicate state.
- **Layout:** fixed left sidebar (~220px) with the screens from Section 4, a top bar with a channel switcher (Personal / TOGA page toggle, visible on every screen that's channel-specific), and a single main content column. No nested scrolling regions.
- **No auto-posting, anywhere, under any setting.** This is a hard product rule, not a default that can be toggled off.
- **Calm over flashy.** No gradients, no "AI sparkle" iconography, no animated loading states beyond a simple spinner. The tone should match a piece of engineering software, not a growth-hacking tool — flat surfaces, generous whitespace, the navy/teal/coral/amber palette from Section 2.
- **Forms are short.** Campaign creation, product entry, and voice profile training should each fit on one screen without scrolling past a single viewport on a laptop.

## 7. Data architecture

```sql
CREATE TABLE channels (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,                  -- "Ibrahim Awad (personal)", "TOGA Academy (page)"
    type TEXT NOT NULL,                  -- 'personal' | 'page'
    linkedin_urn TEXT,                   -- LinkedIn member or organization URN
    access_token TEXT,                   -- encrypted at rest
    refresh_token TEXT,                  -- encrypted at rest
    token_expires_at TIMESTAMP,
    content_mix_ratio TEXT DEFAULT '1:3', -- promo:organic
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE style_profiles (
    id INTEGER PRIMARY KEY,
    channel_id INTEGER REFERENCES channels(id),
    fingerprint_json TEXT,               -- structured style descriptor from Claude
    example_posts_json TEXT,             -- raw reference posts, stored as JSON array
    trained_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE products (
    id INTEGER PRIMARY KEY,
    type TEXT NOT NULL,                  -- 'book' | 'course'
    title TEXT NOT NULL,
    description TEXT,
    key_benefits_json TEXT,
    link TEXT,
    audience TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE sources (
    id INTEGER PRIMARY KEY,
    type TEXT NOT NULL,                  -- 'document' | 'topic' | 'rss'
    title TEXT,
    raw_text TEXT,
    file_path TEXT,
    confidentiality_status TEXT DEFAULT 'pending', -- 'pending' | 'cleared' | 'flagged'
    added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE confidentiality_flags (
    id INTEGER PRIMARY KEY,
    source_id INTEGER REFERENCES sources(id),
    reason TEXT,
    resolved BOOLEAN DEFAULT 0,
    resolved_at TIMESTAMP
);

CREATE TABLE insights (
    id INTEGER PRIMARY KEY,
    source_id INTEGER REFERENCES sources(id),
    insight_text TEXT,
    category TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE campaigns (
    id INTEGER PRIMARY KEY,
    product_id INTEGER REFERENCES products(id),
    channel_id INTEGER REFERENCES channels(id),
    goal TEXT,
    num_posts INTEGER DEFAULT 5,
    start_date DATE,
    end_date DATE,
    status TEXT DEFAULT 'draft',         -- 'draft' | 'active' | 'completed'
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE posts (
    id INTEGER PRIMARY KEY,
    channel_id INTEGER REFERENCES channels(id),
    insight_id INTEGER REFERENCES insights(id),     -- nullable
    campaign_id INTEGER REFERENCES campaigns(id),   -- nullable
    variant_group_id TEXT,               -- groups the 3 variants of one draft together
    variant_number INTEGER,
    draft_text TEXT,
    status TEXT DEFAULT 'draft',         -- 'draft' | 'approved' | 'scheduled' | 'posted' | 'rejected'
    scheduled_time TIMESTAMP,
    posted_time TIMESTAMP,
    linkedin_post_id TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE engagement (
    id INTEGER PRIMARY KEY,
    post_id INTEGER REFERENCES posts(id),
    impressions INTEGER,
    likes INTEGER,
    comments INTEGER,
    shares INTEGER,
    logged_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT
);
```

## 8. Data workflow

Three input types feed one generation step, which feeds one shared queue, which branches to two channels:

1. **Documents** (PDF/DOCX/PPTX), **topics/news** (pasted text or RSS), and **campaign briefs** (a product + a goal) are the three ways content enters the system.
2. Every new source runs through the **confidentiality screen** first — Claude checks for client-identifying detail, named projects, or proprietary figures, and flags anything uncertain before a single insight gets extracted.
3. Cleared sources go to **Claude generation**, which always pulls in the relevant channel's **voice profile** (style fingerprint + example posts) as context, and for campaign briefs, also pulls the relevant **product** record.
4. Output lands in the **shared draft queue** as three variants per insight (or per campaign post), tagged with status `draft`.
5. The **Dashboard** is the only place drafts can move from `draft` to `approved`/`scheduled` — there is no automatic promotion path.
6. The **scheduler** (a background loop, not a separate stateless trigger — see Section 10) checks for posts that are `scheduled` and due, applies the channel's content-mix ratio as a sanity check, and posts via the LinkedIn API, updating status to `posted` and recording the `linkedin_post_id`.
7. Engagement numbers get logged manually against each posted item (v1) and shown on the Analytics screen.

## 9. Integrations & APIs

**Anthropic Claude API**
- Official Python SDK (`anthropic`), API key in environment variables.
- Four distinct prompt templates: insight extraction, post generation (3 variants), style fingerprint extraction, campaign sequence generation. Store these as separate files (see repo structure, Section 10) so they can be tuned independently.

**LinkedIn API**
- Register one developer app in the LinkedIn Developer Portal.
- Request the **"Sign In with LinkedIn using OpenID Connect"** and **"Share on LinkedIn"** products.
- Scopes: `w_member_social` for the personal profile channel, `w_organization_social` for the TOGA page channel (Ibrahim is the page admin, which keeps this out of the more restrictive Marketing Developer Platform partner tier reserved for agencies posting on behalf of *other* companies' pages).
- Use the current `/posts` endpoint, not the deprecated `/ugcPosts`.
- OAuth 2.0 authorization code flow. Access tokens expire in 60 days; store and use refresh tokens.
- **Hard rule:** official OAuth + documented endpoints only. Never cookie-based auth, browser automation, or anything that imitates manual LinkedIn usage — this is both a ToS and an account-safety issue on Ibrahim's actual professional profile and the TOGA company page.

**Document ingestion (optional Drive sync)**
- No Google Drive API integration needed. Install Google Drive for Desktop (or `rclone` on a headless server) to mirror the existing Drive knowledge folder to a local directory, and point the file watcher at that local path. Zero OAuth code required for this piece.

**RSS**
- `feedparser` library, no authentication, used for the topic/news input mode.

**Hosting**
- One small always-on VPS (or a PaaS with a persistent volume, e.g. Fly.io). The app, the SQLite database, and the scheduler loop run together as a single process — this avoids the GitHub Actions ephemeral-disk problem (a stateful queue doesn't survive between Actions runs without an external database, which is unnecessary complexity here).

## 10. Technical architecture & coding structure

**Recommended stack:** FastAPI (backend + routes) + Jinja2 templates + Tailwind CSS via CDN + a small amount of `htmx`/vanilla JS for interactivity (variant switching, approve/reject without full page reloads). This keeps everything in Python and plain HTML — no Node build pipeline, no separate frontend repo — while still giving full visual control to hit the polished, Taplio-grade look described in Section 6.

*Faster, lower-fidelity alternative:* Streamlit could replace the entire frontend layer for a quicker v1 at the cost of visual polish and layout control. Worth knowing this option exists, but the recommendation above is the better fit given the explicit design goals in this brief.

## 11. Login / entry screen

This is not a public marketing page — there are no customers to convert. The entry screen is a single branded login (one password, stored as a hashed value in the environment, not in the database) that leads straight to the Dashboard. Keep it simple: the Tacit name/logo, the tagline, a single password field, nothing else. If TOGA's public website ever wants a page acknowledging that this tool exists, that's a separate, much smaller piece of work — not part of this build.
